# Marketing assets

Hero screenshots for the v0.3 announce. All five are sourced from the V2
Cathedral Bible mockups (`docs/ui/bible/`), which are pixel-accurate HTML
renders of the runtime via Playwright + Chromium at retina (2× device
scale). They are not live `cmux` captures; if you want runtime captures for
a future release, see "Capturing live shots" below.

## Social preview

`social-preview.png` is the 1280 × 640 GitHub social card. Cathedral palette, wordmark left, terminal mockup right showing the read/edit/bash flow, and three preattentive-state dots for `v0.3.0` (idle), `Pi 0.74` (thinking), `MIT` (gold accent).

Upload to GitHub via repo Settings → Social preview. Source HTML at `_social-preview.html`; regenerate with:

```bash
node -e "import('playwright').then(async ({chromium}) => { const b = await chromium.launch(); const c = await b.newContext({ viewport: { width: 1280, height: 640 }, deviceScaleFactor: 1 }); const p = await c.newPage(); await p.goto('file://' + process.cwd() + '/docs/marketing/_social-preview.html', { waitUntil: 'networkidle' }); await p.evaluate(() => document.fonts.ready); await p.waitForTimeout(150); await p.screenshot({ path: 'docs/marketing/social-preview.png' }); await b.close(); })"
```

## Theme provenance

- **Cathedral** is the default theme. Source of truth: `src/themes/cathedral.ts` + `docs/ui/CATHEDRAL_UX_SPEC_V2.md`.
- **Amber CRT** mirrors Mission Control v3 and the Stitch design ref ([SumoCode v0.1 — Amber CRT](https://stitch.withgoogle.com/projects/5385606235875789209)). Source of truth: `src/themes/amber-crt.ts`.
- **Obsidian Temple** is the sacred-tech night theme. Source of truth: `src/themes/obsidian.ts` + `docs/ui/stitch/obsidian-temple/DESIGN.md`.

The Bible HTML mockups use parallel `_assets/tokens-<theme>.css` files so each
mockup renders against the correct palette. Adding a new theme means writing
a new tokens CSS file plus a `scene-active-<theme>.html` and re-running
`pnpm render:bible`.

## Regenerating

```bash
pnpm render:bible           # regenerates docs/ui/bible/renders/*.png
# Then refresh the marketing copies:
cp docs/ui/bible/renders/03-splash.png                          docs/marketing/01-cathedral-splash.png
cp docs/ui/bible/renders/scene-active.png                       docs/marketing/02-cathedral-active.png
cp docs/ui/bible/renders/scene-memory-scriptorium-overlay.png   docs/marketing/03-memory-scriptorium.png
cp docs/ui/bible/renders/scene-active-amber-crt.png             docs/marketing/04-amber-crt.png
cp docs/ui/bible/renders/scene-active-obsidian.png              docs/marketing/05-obsidian-temple.png
```

The mockup HTML lives under `docs/ui/bible/`. Some files are generated by
`scripts/gen-bible-element-*.mjs`; edits to those will be overwritten on the
next `pnpm render:bible`. Hand-written scenes (e.g. `scene-active-amber-crt.html`,
`scene-active-obsidian.html`) are safe to edit directly.

## Capturing live shots

Bible mockups are deliberately frozen and ideal for marketing — they show the
intended runtime layout without live-capture timing artifacts. Live cmux
captures are higher fidelity (real font hinting, real cursor blink, real
focus state) but harder to reproduce deterministically.

For a future release that wants live captures:

1. Open `cmux` at landscape `160 × 40` (or portrait `60 × 100`).
2. Launch SumoCode: `bin/sumocode.sh`.
3. Capture with QuickTime / Cleanshot / OBS at 60 fps, then crop to the
   terminal canvas, export PNG.
4. Re-run with `Ctrl+Shift+T` to cycle themes for shots 4 / 5.

## Why these five shots

The announce is "show what it looks like, tell the truth about what it is,
ask for nothing." These five do the heavy lifting:

1. **Splash** — establishes the visual language in one frame: SUMO cat,
   wordmark, Cathedral palette, the DIVINE INVOCATION input frame.
2. **Active read/edit/bash** — proves the renderer can hold a real working
   session: framed user/SUMO chat blocks, inline tool calls (`[read]`,
   `[edit]`, `[bash]`), sidebar with context + cost meter, footer with
   the active state dot. The basic execution flow without delegation
   wrapping.
3. **Memory Scriptorium** — the most distinctive surface. Sells the
   "persistent memory across sessions" line in the announce thread.
4. **Amber CRT** — proves the theme system isn't just a colour swap.
   Different chrome (`╔╗╚╝═║`), different bullet (`❧` in gold), different
   tab markers (`●`/`○`), different palette philosophy (CRT phosphor).
5. **Obsidian Temple** — the third theme, third visual mood. Sacred-tech
   neons + Egyptian glyphs reinforce that themes can carry real personality.

## Personal data

These mockups have been genericized for public release: no real names, no
employer, no machine-specific paths. The IDENTITY example panel shows
`senior FE · indie operator · London / BST` as a representative profile. The MEMORY example facts (`prefers TypeScript strict`, `pnpm not
npm`, `cmux runtime, libghostty`, etc.) are illustrative system prefs, not
private user data.

If you need to swap the example content for a future release, edit
`scripts/gen-bible-element-7.mjs` (it generates the memory editor + the
scriptorium overlay scene) and re-run `pnpm render:bible`.
