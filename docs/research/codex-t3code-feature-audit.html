SumoCode · Competitive Feature Audit

Codex App & T3 Code,
read through a terminal-native lens.

Both apps are GUI orchestration shells wrapped around agent backends. SumoCode is a retained TUI inside a cmux surface. The features that translate are the workflow primitives — worktree isolation, turn-scoped review, one-shot ship — not the Electron chrome. This audit maps every notable feature to what SumoCode already has, what's partial, and what's worth building.

01 The thesis

Codex and T3 Code both solve the same friction: run agents in parallel without them stomping each other, then review and ship the result without leaving the app. SumoCode already owns the hard part — a real retained renderer and a background-task tool. What's missing is the git-isolation + structured-review loop that turns "an agent edited my files" into "an agent worked in its own worktree and I shipped its diff in one keystroke."

Read the two products differently. Codex app is the mature reference design — its worktree handoff model, review pane with inline comments, and automations are the canonical spec. T3 Code is the lean OSS take — it proves the same primitives (worktree-per-task, one-click commit→push→PR, per-turn diff) ship in a small surface. SumoCode should borrow Codex's model and T3's scope discipline.

02 The two tools
REFERENCE · OpenAI Codex app

codex.app

Electron desktop "command center for agentic coding." Mature, broad, ChatGPT-account gated. The feature spec everyone copies.

  • Three thread modes: Local / Worktree / Cloud
  • Worktree handoff between background & foreground
  • Review pane: inline comments, stage/revert per hunk
  • /review code review → inline findings
  • Commit / push / open PR in-app
  • Automations + thread automations (cron + heartbeat)
  • Integrated terminal scoped per thread (Cmd+J)
  • In-app browser w/ element comments; computer use
  • Voice dictation, image gen/input, pop-out window
  • Skills, MCP, memories shared across app/CLI/IDE
LEAN OSS · T3 Code (Theo / t3-oss)

t3-oss/t3-code

Free, open-source, BYOK GUI front-end for existing agents (Codex now; Claude Code, Cursor, Gemini planned). Proves the primitives in a small surface.

  • Three-panel layout: projects · threads · chat
  • Multi-repo, multi-agent parallelism
  • Native git worktree per task
  • One-click commit → push → create PR
  • Per-turn diff viewer (unified + split)
  • Thread sidebar reflects PR status (open/merged/closed)
  • Custom quick actions (project-scoped shell buttons)
  • Auto-run setup actions on new worktree (npm install)
  • Chat vs plan mode; supervised vs full-access
  • Integrated terminal; skills + headless mode planned

The overlap is the signal. Both independently shipped worktree-per-task, one-shot ship, and structured diff review as their headline features. That convergence is the strongest argument for adopting those three in SumoCode.

04 The feature you liked: worktree + review

Codex's worktree model, decoded

This is the mechanic you responded to. Here's exactly how Codex implements it — and what maps cleanly onto SumoCode's cmux foundation. The default is a plain interactive pane, not a delegated task.

Local / Worktree / Cloud at thread start

Worktree creates a 2nd checkout in $CODEX_HOME/worktrees, detached HEAD, so branches stay clean.

Isolation

Each worktree = own files, shared .git

net-new

bg_task is not required here. The default /sumo:worktree <task> just opens a plain interactive cmux pane running sumocode inside a fresh git worktree — the same openCommandInNewSplit plumbing /sumo:diff already uses. You drive that session directly. bg_task only earns its place when you want to dispatch-and-walk-away and have the orchestrator harvest the result.

The win for SumoCode: /sumo:worktree <task> creates a git worktree under a managed dir, opens a cmux split running sumocode in it, and auto-runs a configured setup action. You review with the existing /sumo:diff + /sumo:review, then /sumo:ship does commit→push→PR. Every piece is plumbing you already own — no delegation layer needed for the interactive path.

05 Sleeper features you didn't ask about

Things buried in the docs that are quietly the most interesting for a power user.

CODEX · underrated

Thread automations

Recurring "wake-up calls" that preserve a thread's context — heartbeat polling, long-running follow-up loops, "check on this every hour and continue." This is more than cron: it's stateful resumption. SumoCode's bg_task only wakes on completion; a scheduled context-preserving wake is genuinely new.

CODEX · underrated

Terminal output as context

Codex can read the integrated terminal's live output — check a dev server's status, refer back to a failed build — without you pasting it. cmux already hosts the terminal; piping a pane's scrollback into the agent's context on request is a small, high-leverage bridge.

T3 · underrated

Quick actions on worktree create

Project-scoped shell buttons (npm run dev) with optional keybindings, and crucially auto-run on new worktree. This is the setup-script primitive that makes worktrees actually usable — a fresh checkout needs install before the agent can build.

T3 · underrated

PR status on the thread list

The thread's sidebar icon reflects open / merged / closed PR state. Tiny touch, big orientation win when juggling several parallel agents. SumoCode's sidebar could surface a git/PR badge per session with one gh poll.

06 Recommended: what to adopt

Ranked by leverage-to-effort, scoped to SumoCode's terminal-native reality. Each builds on primitives you already ship.

PRIORITY 01 · highest leverage

The worktree → review → ship loop

/sumo:worktree <task> creates a git worktree under a managed dir, opens a plain interactive sumocode pane inside it via a cmux split, and auto-runs a configured setup action. You drive that session directly. Review with the existing /sumo:diff + /sumo:review, then /sumo:ship commits, pushes, and opens the PR.

Why first: it's the convergent feature both products lead with, it's exactly what you flagged, and every piece (cmux splits via openCommandInNewSplit, hunkdiff, review scribe, gh) already exists. No bg_task needed — the interactive pane is the same plumbing /sumo:diff uses. This is assembly, not invention.

PRIORITY 02 · strong follow-up

Handoff + an optional detached flavor

Two things here. (a) Handoff: git glue to relocate a worktree's branch into the main session for inspection, or push a foreground thread into a background worktree — Codex's value is the safe one-branch-per-worktree enforcement. (b) An opt-in /sumo:worktree --detach that routes through bg_task's sumocode runner for the genuine dispatch-and-walk-away / fan-out-N-agents case, where the orchestrator harvests results.

Why: the interactive pane (P1) covers the common case; bg_task is the right tool only when you want programmatic result collection, not babysitting.

PRIORITY 03 · self-contained quick win

Inline image rendering in the transcript

Pi ships an Image TUI component that renders in Kitty / iTerm2 / Ghostty / WezTerm — and cmux embeds libghostty, so it qualifies. SumoTUI's retained renderer never wired it up: ChatBlock has no image variant, and blocksFromContentPart hits its default: case for type: "image" parts and drops them. Add an image block type, carry the base64 + MIME through the view-model, and render via Pi's Image in the scroll-renderer.

Why: pasted/generated images currently vanish in default mode — Pi's classic renderer would have shown them. No new dependency (the component is Pi's), no protocol work, no git surface. Pure assembly inside src/sumo-tui/transcript.

PRIORITY 04 · editor polish · OpenCode-style

Show [Image 1] in the editor on image paste

Pi's handleClipboardImagePaste() writes the clipboard image to a temp file and does insertTextAtCursor(filePath) — so the composer shows a raw /var/folders/…/pi-clipboard-<uuid>.png path as literal text. OpenCode instead inserts a readable [Image #1] token, keeps a side-map of token→path, and expands tokens back into real attachments on submit.

Why · distinct from P3: P3 renders a sent image in the transcript; this keeps the input legible while composing and lets you stack several. Touches three points — intercept onPasteImage, hold token→path in editor-draft-state.ts, expand on submit. Medium effort: it overrides Pi-default behavior, so the non-SumoTUI path must stay intact.

PRIORITY 05 · cheap, high-orientation

Git/PR badge in the sidebar + quick actions

Surface per-session branch + PR status (open/merged/closed) in the sidebar via a cached gh poll. Add project-scoped quick actions (configurable shell buttons, auto-run on worktree create) — the T3 setup-script primitive.

Why: small surface, big day-to-day clarity when several agents run in parallel.

PRIORITY 06 · exploratory

Stateful scheduled automations

Context-preserving thread wake-ups (cron + heartbeat) on top of bg_task. Plus a "read this cmux pane's output into context" bridge. Genuinely new capability, but heavier and less requested — stage it after the core loop proves out.

Why last: highest novelty, lowest certainty of payoff; revisit once the worktree loop is daily-driven.

07 What to skip

Features that don't translate to a terminal-native tool, or that SumoCode already covers better elsewhere.

FeatureWhy skip
Pop-out / always-on-top windowGUI-only affordance. cmux/tmux pane management already covers spatial arrangement.
In-app browser + element commentsHeavy. agent_browser + chrome-devtools MCP already give SumoCode real browser control; element-comment UX isn't worth a custom renderer.
Computer use (drive macOS apps)Large surface, narrow payoff, region-restricted even in Codex. Out of scope for a coding TUI.
Image generation in-threadNice-to-have, not workflow-critical. Pi handles image input already; generation is a model/provider concern.
Cloud thread modeRequires hosted infra. SumoCode is local-first by design; bg_task + worktrees give the parallelism without a backend.
Voice dictationOS-level dictation already exists; low marginal value inside the TUI.
Three-panel Electron layoutSumoTUI's sidebar + transcript + footer is the terminal-native equivalent and is already the house design.

The discipline here mirrors T3's: don't rebuild Electron chrome in a terminal. Adopt the workflow primitives that convergent products validated, lean on MCP for the browser/vision surface, and keep SumoCode's identity — a fast, retained, cmux-native renderer.
