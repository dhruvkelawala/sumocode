## Round 2: post-#53 regression (#54)

- **Mode 1 — offline/temp agent (pre-fix control):** 0.74% avg CPU, 24.6% peak, 4 bytes/sec stdout, 0.00 frame-scheduler renders/sec.
- **Mode 2 — full extensions (pre-fix):** 0.75% avg CPU, 29.8% peak, 79 bytes/sec stdout, 1.32 setTimeout fires/sec, 0.03 setInterval fires/sec. The additional hot stacks were `src/sidebar.ts` → `src/memory.ts` Remnic recall retries plus the MCP health-check interval.
- **Typing burst:** `sumocode54` sent through the PTY during the 30s sample. Pre-fix full mode typing-window CPU averaged 0.53% and peaked at 2.10%; no render loop appeared.
- **Diagnosis:** the #53 frame scheduler fix held. The new full-mode idle tax was the sidebar memory cache scheduling an empty prompt on `session_start`, then retrying Remnic every 5s and calling `requestRender()` even while the splash had no messages/sidebar.
- **Fix:** do not schedule memory recall for the empty splash prompt; ignore empty scheduled prompts; only call the sidebar render callback when the memory snapshot actually changes. Retry probes still continue for real prompts so Remnic recovery is detected.
- **Post-fix full extensions:** 0.56% avg CPU, 24.9% peak, 76 bytes/sec stdout during the typing-included 30s sample, 0.33 setTimeout fires/sec, 0.03 setInterval fires/sec, 0.00 frame-scheduler renders/sec. The Remnic retry stacks disappeared; remaining timers were Pi TUI input renders from the typing burst plus MCP health checks.

Conclusion: no retained render loop is present under full extensions. The actionable regression was full-mode sidebar memory work happening on the splash/empty prompt path; that idle work is now removed.
