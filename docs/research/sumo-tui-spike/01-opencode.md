## 1. Repository overview

OpenCode is a Bun and TypeScript monorepo: the root manifest declares ESM and Bun 1.3.13, its root `dev` command runs `packages/opencode` with Bun browser conditions, and the workspace list includes package globs for `packages/*`, console packages, the JavaScript SDK, and Slack support. (`package.json:3-9`, `package.json:22-29`) The same root manifest pins `@opentui/core` and `@opentui/solid` at catalog version 0.1.105, and the terminal app imports those packages directly in its renderer and session code. (`package.json:37-38`, `packages/opencode/src/cli/cmd/tui/app.tsx:1-4`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:24-25`) The `packages/opencode` package is the CLI package: it publishes an `opencode` binary, runs its development entry with `bun run --conditions=browser ./src/index.ts`, and contains the TUI implementation under `src/cli/cmd/tui`. (`packages/opencode/package.json:16-24`)

The TUI entry path is a threaded CLI command. `TuiThreadCommand` is registered as the default command, resolves project and prompt arguments, creates or connects to a worker transport, validates a requested session, then calls `tui(...)` with URL, fetch, event source, config, current directory, and parsed CLI args. (`packages/opencode/src/cli/cmd/tui/thread.ts:74-109`, `packages/opencode/src/cli/cmd/tui/thread.ts:187-249`) The `tui(...)` function creates the OpenTUI renderer, waits for terminal theme mode, and renders a Solid provider tree into that renderer. (`packages/opencode/src/cli/cmd/tui/app.tsx:108-136`, `packages/opencode/src/cli/cmd/tui/app.tsx:136-199`) The provider tree is dense but explicit: args, exit, KV, toast, route, config, SDK, project, sync, theme, local state, keybinds, prompt stash, dialog, command, frecency, prompt history, prompt ref, and editor contexts all wrap the main app. (`packages/opencode/src/cli/cmd/tui/app.tsx:143-190`)

The session screen lives in `routes/session/index.tsx`, and it imports both OpenTUI core renderables and OpenTUI Solid hooks directly. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1-24`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:52-53`) That file owns the chat layout, sidebar layout, in-app scrollback, message rendering, session commands, permission and question prompts, and the bottom prompt attachment. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:119-180`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1037-1230`) The input editor is a separate component under `component/prompt/index.tsx`, and it imports OpenTUI `TextareaRenderable`, `MouseEvent`, `PasteEvent`, and `decodePasteBytes`. (`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:1-29`) This split is important for SumoCode: the session route owns transcript navigation and viewport policy, while the prompt component owns editing, paste, autocomplete, history, submission, and prompt extmarks. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:300-351`, `packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:521-597`)

## 2. Altscreen and viewport ownership

OpenCode enters OpenTUI through `createCliRenderer(rendererConfig(input.config))`. (`packages/opencode/src/cli/cmd/tui/app.tsx:68-89`, `packages/opencode/src/cli/cmd/tui/app.tsx:133-134`) The OpenCode renderer config sets passthrough external output, 60 FPS target, disabled Ctrl-C auto-exit, Kitty keyboard support, disabled auto-focus, mouse enablement, and console copy handling; it does not set a `screenMode` property in the returned config object. (`packages/opencode/src/cli/cmd/tui/app.tsx:68-88`) OpenTUI defaults `screenMode` to `alternate-screen` when no screen mode is supplied, and it documents the available modes as `alternate-screen`, `main-screen`, and `split-footer`. (`opentui/packages/core/src/renderer.ts:282-289`, `opentui/packages/core/src/renderer.ts:188-195`)

The actual terminal ownership is in OpenTUI native setup. The TypeScript renderer calls `setupTerminal` with `this._screenMode === alternate-screen`, and the Zig renderer enters alt screen when `useAlternateScreen` is true. (`opentui/packages/core/src/renderer.ts:2487-2498`, `opentui/packages/core/src/zig/renderer.zig:392-407`) The ANSI constants are the standard DEC private mode sequences `CSI ? 1049 h` and `CSI ? 1049 l`. (`opentui/packages/core/src/zig/ansi.zig:178-179`) Cleanup runs through `renderer.destroy()`: OpenCode clears the terminal title and destroys the renderer in its exit context, while OpenTUI removes signal and stdin listeners, restores raw mode, restores stdout, destroys the render tree, clears queues, and destroys the native renderer. (`packages/opencode/src/cli/cmd/tui/context/exit.tsx:32-50`, `opentui/packages/core/src/renderer.ts:3473-3485`, `opentui/packages/core/src/renderer.ts:3487-3549`, `opentui/packages/core/src/renderer.ts:3556-3621`)

This means OpenCode does not depend on the host terminal scrollback for chat history while in altscreen. The live transcript is a normal OpenTUI subtree inside the alternate screen, and the terminal scrollback outside the alternate buffer is irrelevant to paging current chat content. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1055-1075`, `opentui/packages/core/src/zig/ansi.zig:178-179`) OpenTUI also has split-footer and `writeToScrollback` APIs, but those require `screenMode` `split-footer` plus `externalOutputMode` `capture-stdout`, whereas OpenCode sets `externalOutputMode` to passthrough and leaves screen mode at OpenTUI default alt screen. (`opentui/packages/core/src/renderer.ts:1454-1459`, `opentui/packages/core/src/renderer.ts:1798-1801`, `packages/opencode/src/cli/cmd/tui/app.tsx:71-80`) For SumoCode, the direct lesson is that altscreen is compatible with rich chat scrollback if the application, not the terminal, owns the transcript viewport. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1058-1075`, `opentui/packages/core/src/renderables/ScrollBox.ts:58-72`)

## 3. Chat scrollback and pager

The core chat viewport is a single `<scrollbox>` in the session route. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1058-1075`) It stores a `ScrollBoxRenderable` ref, applies optional scrollbar padding, configures vertical scrollbar colors and visibility from theme plus KV state, enables `stickyScroll`, sets `stickyStart` to `bottom`, gives the scrollbox `flexGrow={1}`, and passes a configured scroll acceleration object. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:241-243`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1058-1075`) The transcript children are a one-line spacer followed by a Solid `<For each={messages()}>` loop that renders user messages, assistant messages, and revert markers into the scrollbox. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1076-1172`) The prompt, permission prompt, question prompt, and subagent footer are outside the scrollbox in a fixed bottom box, so paging the transcript cannot move the input editor. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1173-1205`)

OpenTUI `ScrollBoxRenderable` is purpose-built for this. Its options include `stickyScroll`, `stickyStart`, independent X/Y scrolling, scroll acceleration, and viewport culling. (`opentui/packages/core/src/renderables/ScrollBox.ts:58-72`) Its constructor creates a wrapper, a viewport with `overflow: hidden`, a translated content box, and vertical and horizontal scrollbars. (`opentui/packages/core/src/renderables/ScrollBox.ts:318-387`) When vertical scroll position changes, OpenTUI translates the content by negative position and marks manual scroll only if content is meaningfully scrollable and the user has left the sticky position. (`opentui/packages/core/src/renderables/ScrollBox.ts:366-385`) When content or viewport size changes, sticky mode reapplies the configured edge if the user has not manually scrolled away, which is exactly the behavior a streaming chat wants. (`opentui/packages/core/src/renderables/ScrollBox.ts:752-781`)

OpenCode uses two explicit bottom-snap paths on top of the sticky behavior. After session sync, it calls `scroll.scrollBy(100_000)` if the route still matches the loaded session, which is a brute-force jump to the bottom after async bootstrap. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:184-212`) It also defines `toBottom()` as a delayed `scroll.scrollTo(scroll.scrollHeight)` and wires that to session changes and prompt submission. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:346-351`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1034-1035`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1193-1201`) The delayed call is defensive because layout and message height can update after reactive state changes; the code explicitly waits 50 ms before measuring `scrollHeight`. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:346-350`)

Keyboard paging is command-driven rather than hardcoded in the scrollbox. The session command registry defines page up and page down as half-screen scrolls, line up and line down as single-line scrolls, quarter-page scrolls, first and last message jumps, last-user-message jump, and next/previous visible-message navigation. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:686-820`) Defaults bind page up to `pageup,ctrl+alt+b`, page down to `pagedown,ctrl+alt+f`, first to `ctrl+g,home`, and last to `ctrl+alt+g,end`. (`packages/opencode/src/config/keybinds.ts:40-49`) The command provider listens globally for keypresses when no dialog is open, checks registered command keybinds, prevents default, and invokes the command’s `onSelect`. (`packages/opencode/src/cli/cmd/tui/component/dialog-command.tsx:63-75`) OpenTUI’s internal key handler intentionally runs global handlers before focused-renderable handlers, so a command can consume a key before the textarea processes it. (`opentui/packages/core/src/lib/KeyHandler.ts:130-180`)

Message-boundary navigation uses rendered child positions, not data indices alone. `findNextVisibleMessage` asks the scrollbox for children, filters those whose IDs correspond to messages containing non-synthetic non-ignored text parts, sorts them by rendered `y`, and finds the next or previous boundary relative to the viewport. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:300-329`) `scrollToMessage` either pages by one viewport height when no target exists or scrolls by the delta between the child’s rendered `y` and the scrollbox’s rendered `y`. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:331-344`) The same rendered-child lookup powers timeline jump, fork-from-timeline jump, and last-user-message jump. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:443-463`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:474-486`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:773-803`)

One critical limitation is that this is not unbounded history. Initial session sync requests only the latest 100 messages, and the live `message.updated` reducer trims a session’s message array back to 100 by shifting the oldest message and deleting its parts. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:499-520`, `packages/opencode/src/cli/cmd/tui/context/sync.tsx:234-270`) Therefore OpenCode’s in-app scrollback means an app-owned viewport over the loaded transcript window, not infinite terminal scrollback. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:253-269`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1077-1172`) If SumoCode needs multi-thousand-message sessions, it should steal the scrollbox model but add virtualization or paged transcript loading rather than blindly copying the 100-message cap. (`opentui/packages/core/src/renderables/ScrollBox.ts:38-53`, `packages/opencode/src/cli/cmd/tui/context/sync.tsx:501-504`)

## 4. Input editor

(`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:89-112`, `packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:203-218`) OpenCode exposes a `PromptRef` with focus, blur, set, reset, and submit methods; the session screen captures that ref and stores it in a prompt-ref context. (`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:494-504`) It also updates `input.traits` so autocomplete can capture escape, navigation, submit, and tab keys, while disabled or shell mode can suspend normal editor behavior. (`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:506-519`, `opentui/packages/core/src/renderables/EditBufferRenderable.ts:13-19`, `opentui/packages/core/src/renderables/EditBufferRenderable.ts:261-270`)

The textarea is rendered with placeholder text, muted leader-mode colors, `minHeight={1}`, `maxHeight={6}`, custom keybindings, content-change synchronization, a double-deferred submit path, bracketed paste handling, mouse focus, focused background, cursor color, and syntax style. (`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:1037-1051`, `packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:1129-1232`) OpenTUI’s textarea merges default editor bindings with custom bindings, maps actions to editing commands, inserts printable text when no modifier key is active, and delegates paste to decoded bytes after stripping ANSI sequences when the application does not prevent default. (`opentui/packages/core/src/renderables/Textarea.ts:58-126`, `opentui/packages/core/src/renderables/Textarea.ts:187-191`, `opentui/packages/core/src/renderables/Textarea.ts:218-296`) OpenCode maps its config schema to OpenTUI textarea actions with names such as `input_delete_to_line_end`, and it prepends return-to-submit plus meta-return-to-newline defaults. (`packages/opencode/src/cli/cmd/tui/component/textarea-keybindings.ts:6-72`)

The editor supports rich prompt parts through extmarks. (`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:521-563`, `packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:157-225`) Before submit, it expands pasted text parts inline, filters out text parts that have already been expanded, preserves file and agent parts, adds editor-selection synthetic context when present, and then calls shell, slash command, or prompt APIs. (`packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx:833-855`) Prompt history is persisted as JSONL under `prompt-history.jsonl`, capped at 50 entries, self-heals corrupt lines by rewriting valid entries, and appends or rewrites on new history entries. Autocomplete is also an OpenTUI layout, not terminal-native completion. (`packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:91-129`, `packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:131-157`, `packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:227-428`) When autocomplete opens, it suspends command keybinds, and when it hides, it re-enables them. (`packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:523-543`) The popup is an absolute box above the prompt with z-index 100 and its own `scrollbox` with hidden scrollbar and the same scroll-acceleration config. (`packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:653-680`) Keyboard and mouse selection both update the selected option and call `select()`, so the completion UI is fully in-app. (`packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:596-649`, `packages/opencode/src/cli/cmd/tui/component/prompt/autocomplete.tsx:689-719`)

## 6. Mouse handling

Mouse capture is configurable in OpenCode and implemented by OpenTUI. The TUI schema exposes `mouse` as an optional boolean with default true behavior, and `rendererConfig` combines that with `OPENCODE_DISABLE_MOUSE` before passing `useMouse` to OpenTUI. (`packages/opencode/src/cli/cmd/tui/config/tui-schema.ts:14-27`, `packages/opencode/src/cli/cmd/tui/app.tsx:68-80`) OpenTUI’s terminal layer enables normal mouse tracking, button-event tracking, optional any-event tracking, and SGR mouse mode with DEC private mode sequences. (`opentui/packages/core/src/zig/ansi.zig:181-188`, `opentui/packages/core/src/zig/terminal.zig:584-609`) It also force-disables mouse tracking, bracketed paste, focus tracking, and alt screen on reset. (`opentui/packages/core/src/zig/terminal.zig:170-198`)

OpenTUI routes mouse through a hit grid. For scroll events, it hit-tests the pointer, falls back to the focused renderable, and dispatches the scroll to that target. (`opentui/packages/core/src/renderer.ts:2827-2842`) For left-button down on selectable renderables, it starts a selection; drag and up update or finish selection; and ordinary down, over, out, drag, drop, and up events dispatch through renderable mouse handlers. (`opentui/packages/core/src/renderer.ts:2850-2895`, `opentui/packages/core/src/renderer.ts:2905-2974`) OpenCode turns off OpenTUI auto-focus in renderer config, which prevents arbitrary left clicks from automatically focusing the nearest focusable renderable. (`packages/opencode/src/cli/cmd/tui/app.tsx:75-80`, `opentui/packages/core/src/renderer.ts:2786-2795`)

The transcript scrollbox gets native wheel behavior from `ScrollBoxRenderable`: wheel direction maps to scrollTop or scrollLeft, shift swaps axes, acceleration is applied, fractional deltas accumulate, and manual scroll state is set when there is meaningful scrollable content. (`opentui/packages/core/src/renderables/ScrollBox.ts:540-587`) OpenCode adds app-level copy behavior: the root box either copies selection on mouse-up or, in the experimental Windows-style mode, copies on right-click and handles Ctrl-C or Esc selection dismissal. (`packages/opencode/src/cli/cmd/tui/app.tsx:256-289`, `packages/opencode/src/cli/cmd/tui/app.tsx:837-850`) Dialog overlays repeat the same copy-on-select pattern at a high z-index. (`packages/opencode/src/cli/cmd/tui/ui/dialog.tsx:154-181`) Tool blocks and user messages guard click handlers by checking whether a text selection exists, so a mouse-up after selecting text does not also open a message dialog or toggle expansion. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1144-1156`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1690-1698`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1753-1768`)

## 7. State management

The TUI state model is Solid stores plus SDK events. `SDKProvider` creates an OpenCode SDK client with base URL, abort signal, directory, fetch, and headers, then exposes a Solid global emitter. (`packages/opencode/src/cli/cmd/tui/context/sdk.tsx:21-38`) Incoming global events are queued and flushed in a Solid `batch` so multiple event-driven store updates can produce one render. (`packages/opencode/src/cli/cmd/tui/context/sdk.tsx:40-58`) If no external event source is supplied, the provider starts an SSE loop with retry backoff and drains queued events before reconnecting. (`packages/opencode/src/cli/cmd/tui/context/sdk.tsx:74-108`) Cleanup aborts the SDK and SSE controllers and clears timers. (`packages/opencode/src/cli/cmd/tui/context/sdk.tsx:126-130`)

`useEvent` filters SDK global events by directory or workspace and ignores sync payloads before handing them to UI subscribers. (`packages/opencode/src/cli/cmd/tui/context/event.ts:9-31`) `SyncProvider` declares one Solid store for providers, agents, commands, permissions, questions, sessions, statuses, diffs, todos, messages, parts, LSP, MCP, formatter, and VCS state. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:37-105`) Its event reducer updates permission queues, question queues, todos, session diffs, session records, statuses, messages, parts, and deltas in sorted arrays or keyed maps. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:114-350`) Bootstrap loads blocking provider, model, agent, config, project, and optional session-list data first, then loads non-blocking sessions, commands, LSP, MCP, resources, formatter, statuses, auth, VCS, and workspace data before marking status complete. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:356-444`)

This state model is why the scrollbox can be declarative. The session route’s `messages` memo is just `sync.data.message[route.sessionID] ?? []`, and the scrollbox’s children are derived from that memo. (`packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:136-149`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1077-1172`) Streaming assistant text arrives as `message.part.delta` events that append to the matching part field, and Solid re-renders the affected `TextPart`, `ReasoningPart`, or tool component. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:308-323`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1447-1511`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:1516-1600`) SumoCode should copy this separation: transport events mutate normalized state, while scroll position and viewport policy stay in the route component. (`packages/opencode/src/cli/cmd/tui/context/sync.tsx:463-524`, `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:240-351`)

## 8. Theme and styling

Themes are first-class TUI data. The theme context imports a large bundled theme set, stores plugin and custom themes, resolves references through `defs`, and returns RGBA values for all theme slots. (`packages/opencode/src/cli/cmd/tui/context/theme.tsx:1-47`, `packages/opencode/src/cli/cmd/tui/context/theme.tsx:88-122`, `packages/opencode/src/cli/cmd/tui/context/theme.tsx:132-155`, `packages/opencode/src/cli/cmd/tui/context/theme.tsx:198-255`) The default OpenCode theme defines dark and light ramps, semantic colors, text colors, background slots, border slots, diff colors, markdown colors, and syntax colors. (`packages/opencode/src/cli/cmd/tui/context/theme/opencode.json:3-43`, `packages/opencode/src/cli/cmd/tui/context/theme/opencode.json:80-151`, `packages/opencode/src/cli/cmd/tui/context/theme/opencode.json:152-160`) The context can also generate a system theme from the terminal palette, including transparent background support, ANSI-derived semantic colors, diff backgrounds, markdown colors, and syntax colors. (`packages/opencode/src/cli/cmd/tui/context/theme.tsx:516-624`)

The renderer background is updated from the active theme, and syntax styles are generated as `SyntaxStyle` objects for normal and subtle thinking text.
