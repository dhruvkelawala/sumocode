# Deep dive: `benvinegar/opentui-island` integration patterns

This spike reads `opentui-island` as a practical bridge between Node-hosted TUIs and OpenTUI-rendered React islands. The package describes itself as embedding OpenTUI islands inside Node terminal UIs such as `pi-tui` and Ink, via a Bun sidecar (`package.json:4`, `README.md:1-3`). The decision lens here is SumoCode/sumo-tui: which patterns are worth adopting, and what would be required to remove the sidecar. OpenTUI-internal citations in section 7 refer to installed `node_modules/@opentui/core` files for the exact dependency version declared by this repo (`package.json:74-77`).

## 1. Architecture overview

The project is deliberately split into a Node host side and a Bun render side. The README states the design goal directly: “Keep your host app in Node,” render the embedded island in “a local Bun sidecar,” and reuse one island module across `pi-tui`, Ink, and lower-level hosts (`README.md:8-12`). The runtime requirements match that split: Node 18+, Bun 1.3+, and React are required (`README.md:16-20`), while `package.json` encodes both `node >=18` and `bun >=1.3.10` in `engines` (`package.json:109-112`).

The host abstraction is `OpenTuiHost`. It exposes the operations a foreign TUI needs: `mount`, `updateProps`, bridge `onEvent`/`waitForEvent`/`sendCommand`, `resize`, `focus`/`blur`, `sendKey`, `sendMouse`, `renderFrame`, and `destroy` (`src/core/host.ts:10-35`). The shared controller wraps that host and stores one current island plus one cached frame (`src/core/controller.ts:35-44`). Adapters are thin: `createPiTuiSurface` and `InkSurface` either create their own controller/host or consume a controller supplied by the caller (`docs/api.md:35-45`).

The IPC mechanism is JSON-lines over stdio pipes, not sockets or shared memory. The Node client spawns `bun server.js` through `node:child_process.spawn`, with `stdio: ["pipe", "pipe", "pipe"]` (`src/sidecar/client.ts:474-487`). Requests are serialized with `JSON.stringify({ id, method, ...params })` and written to the child’s stdin with a newline (`src/sidecar/client.ts:304-335`). Responses and island events are read line-by-line from child stdout with `node:readline` (`src/sidecar/client.ts:92-95`, `src/sidecar/client.ts:252-264`). On the Bun side, `server.ts` reads newline-delimited JSON requests from `process.stdin`, parses them, executes `handleRequest`, and writes JSON responses or event messages to `process.stdout` (`src/sidecar/server.ts:40-45`, `src/sidecar/server.ts:279-301`).

The sidecar does the actual OpenTUI work: it dynamically imports the island module, wraps it in `OpenTuiIslandBridgeProvider`, mounts it into an offscreen OpenTUI host, and returns captured frames (`src/sidecar/server.ts:110-170`, `src/sidecar/server.ts:189-223`, `src/sidecar/server.ts:261-263`). That offscreen host is backed by `@opentui/core/testing`’s `createTestRenderer` and `@opentui/react`’s `createRoot` (`src/sidecar/offscreen-host.ts:1-9`, `src/sidecar/offscreen-host.ts:123-137`).

The sidecar is per `OpenTuiHost`/controller, not global. `createOpenTuiSidecarHost` spawns one child process and returns one `SidecarOpenTuiHost` (`src/sidecar/client.ts:474-499`). The server has one process-global `host` and one `loadedIsland`, and each `mount` replaces `loadedIsland` (`src/sidecar/server.ts:26-38`, `src/sidecar/server.ts:220-223`). Therefore the default `createPiTuiSurface` path creates a controller, which creates one host/sidecar for that surface (`src/adapters/pi-tui/index.ts:503-534`); a caller can share a controller, but one sidecar still has one active mounted island at a time (`src/core/controller.ts:37-44`, `src/sidecar/server.ts:220-223`).

## 2. Surface bridge code

The pi-tui adapter is centered on `PiTuiOpenTuiSurface`, which explicitly implements `Component` and `Focusable` (`src/adapters/pi-tui/index.ts:122-124`). Its creation options require a fixed `height`, accept either an `island` or an existing `controller`, and allow a `requestRender` callback plus `initialWidth` (`src/adapters/pi-tui/index.ts:27-35`). That class is the Surface bridge: it translates pi-tui’s `render(width): string[]` model into OpenTUI frame syncs, ANSI-line caching, and input forwarding.

The constructor receives a controller, fixed height, initial width, and optional render callback (`src/adapters/pi-tui/index.ts:137-142`). It stores `height` in a readonly field, normalizes `initialWidth`, initializes a blank cache of exactly `height` lines, and immediately blurs the controller in the background (`src/adapters/pi-tui/index.ts:143-149`). The class also exposes readiness from the controller: `ready`, `readyState`, `readyError`, and `waitUntilReady()` (`src/adapters/pi-tui/index.ts:163-187`).

`createPiTuiOpenTuiSurface` is the factory that turns an island/controller into a pi-tui component. It normalizes `initialWidth`, requires an island unless the controller already has one, and creates a controller when one was not injected (`src/adapters/pi-tui/index.ts:503-527`). When the factory creates the controller, it passes a `size` object with `width: initialWidth` and `height: options.height`, plus keyboard and ready callbacks (`src/adapters/pi-tui/index.ts:513-527`). It then instantiates `PiTuiOpenTuiSurface` with the controller, fixed height, initial width, and render callback (`src/adapters/pi-tui/index.ts:529-534`). The public alias `createPiTuiSurface` is simply `createPiTuiOpenTuiSurface` (`src/adapters/pi-tui/index.ts:546`).

Frames flow back through `sync()` and `render()`. `sync(width)` stores a pending normalized width and starts a serialized sync loop if one is not already running (`src/adapters/pi-tui/index.ts:222-234`). The loop collapses multiple width requests to the latest width and calls `controller.syncFrame({ width, height: this.height })` (`src/adapters/pi-tui/index.ts:211-218`). The controller treats resize and frame fetch as one operation: if a size is supplied, it resizes the host before requesting `renderFrame()` (`src/core/controller.ts:237-246`). The surface then `applyFrame()`s the returned `HostFrame`: it diffs against the cached frame, converts full frames with `hostFrameToAnsiLines`, converts changed lines with `hostLineToAnsi`, truncates to the render width, and calls `requestRender()` when there was a repaint or cursor change (`src/adapters/pi-tui/index.ts:193-208`). Finally, `render(width)` returns the cached lines normalized to the requested width and fixed height, while kicking off a background sync when the width changed or no frame is cached (`src/adapters/pi-tui/index.ts:359-365`).

Key input forwarding is intentionally gated by focus. The `focused` setter stores `_focused` and calls `controller.focus()` or `controller.blur()` in the background (`src/adapters/pi-tui/index.ts:175-181`). `sendInput(data)` exits early when not focused, otherwise sends `{ sequence: data }` through `controller.sendKey()` and syncs the frame (`src/adapters/pi-tui/index.ts:290-298`). The general `handleInput(data)` path first tries to parse SGR mouse input, dispatches mouse if present, and otherwise forwards raw input through `sendInput()` (`src/adapters/pi-tui/index.ts:343-350`). `wantsKeyRelease = true` tells pi-tui this focusable wants release events too (`src/adapters/pi-tui/index.ts:123-124`).

Mouse routing depends on explicit screen bounds because pi-tui’s render contract only supplies width. The surface can store `PiTuiScreenBounds` with row, column, optional width, and optional height (`src/adapters/pi-tui/index.ts:64-69`, `src/adapters/pi-tui/index.ts:151-157`). Mouse input is parsed with `parseSgrMouseInput`, checked against bounds, optionally focuses the surface, and is translated from terminal coordinates to surface-local coordinates before `sendMouse()` (`src/adapters/pi-tui/index.ts:310-340`). `attachPiTuiMouseSupport` enables SGR mouse mode on the terminal, adds a TUI input listener, focuses the surface when an inside-bounds mouse event arrives, and returns a disposer that detaches the listener and disables mouse mode (`src/adapters/pi-tui/index.ts:375-399`). The SGR parser itself converts terminal coordinates from one-based to zero-based and maps button/motion/scroll/modifier bits into `HostMouseInput` (`src/core/terminal-mouse.ts:28-87`).

The modal helper, `createPiTuiOpenTuiModal`, builds on the same surface but owns focus, optional mouse support, close-event waiting, timeout handling, and teardown (`src/adapters/pi-tui/index.ts:402-411`). It creates a surface using the TUI’s terminal columns and `requestRender`, focuses it by default, attaches mouse support unless disabled, and creates a result promise that resolves when a bridge event type is in `closeOn` (`src/adapters/pi-tui/index.ts:412-462`). Its `waitForResult()` always destroys the surface afterward (`src/adapters/pi-tui/index.ts:485-498`). There is no backdrop or dimming implementation in this helper; it returns `surface` plus lifecycle methods, so visual overlay/backdrop behavior belongs to the host overlay that adds the component (`src/adapters/pi-tui/index.ts:485-498`).

## 3. Frame transport

A `HostFrame` is a rectangular terminal frame: width, height, an array of `HostLine`, and optional cursor state (`src/core/types.ts:24-30`). Each line is an array of styled spans (`src/core/types.ts:12-15`). A span includes text, display width, optional foreground/background colors, and bold/italic/underline flags (`src/core/types.ts:1-10`). Host input types are similarly minimal: key input is just a raw sequence, and mouse input is local x/y plus event type, button or scroll direction, and modifier booleans (`src/core/types.ts:32-51`).

Frames are serialized as JSON response payloads. The sidecar protocol defines request methods for `resize`, `focus`, `blur`, `sendKey`, `sendMouse`, `renderFrame`, and `destroy`; the response union allows an `ok: true` result of `HostFrame` or handshake data (`src/sidecar/protocol.ts:14-82`). The client serializes each request to one JSON line and writes it to stdin (`src/sidecar/client.ts:304-335`). The sidecar parses each line into an `OpenTuiSidecarRequest`, executes the method, and writes `JSON.stringify({ id, ok: true, result })` on success (`src/sidecar/server.ts:279-309`). For frame sync specifically, the `renderFrame` request returns `await ensureHost().renderFrame()` (`src/sidecar/server.ts:261-263`), and the client exposes that as `request<HostFrame>("renderFrame")` (`src/sidecar/client.ts:447-449`).

The ANSI conversion layer is deliberately tiny. `hostSpanToAnsi()` converts a span’s style to SGR codes: bold `1`, italic `3`, underline `4`, foreground `38;2;r;g;b`, and background `48;2;r;g;b` (`src/core/ansi.ts:18-43`). A styled span is emitted as `ESC[...m + text + reset`, while an unstyled span is emitted as plain text (`src/core/ansi.ts:46-54`). A host line joins span ANSI, and a host frame maps every line through that function (`src/core/ansi.ts:56-64`). The pi-tui adapter then truncates ANSI lines to width with pi-tui’s `truncateToWidth` (`src/adapters/pi-tui/index.ts:109-119`, `src/adapters/pi-tui/index.ts:193-203`).

Frame diffing happens in the host adapter, not in the sidecar protocol. `diffHostFrames(previous, next)` forces a full repaint when there is no previous frame or when width, height, or line count changed (`src/core/frame-diff.ts:63-82`). Otherwise it compares each line span-by-span, where equality includes text, width, colors, and text attributes (`src/core/frame-diff.ts:19-49`, `src/core/frame-diff.ts:85-100`). The diff also reports cursor changes through a cursor equality check (`src/core/frame-diff.ts:51-60`, `src/core/frame-diff.ts:94-100`). `PiTuiOpenTuiSurface.applyFrame()` uses that diff to either rebuild all cached ANSI lines or patch only changed rows (`src/adapters/pi-tui/index.ts:193-208`).

Bridge events are a separate JSON message path from frames. A bridge event is `{ type, payload }`, where payload must be an `OpenTuiIslandValue` (`src/core/bridge.tsx:4-12`, `src/core/island.ts:4-15`). The server writes island-originated events as `{ event }` JSON lines (`src/sidecar/server.ts:40-45`, `src/sidecar/server.ts:70-73`). The client distinguishes event messages from request responses with `isOpenTuiSidecarEventMessage`, dispatches listeners, and resolves matching waiters (`src/sidecar/client.ts:252-264`, `src/sidecar/client.ts:201-249`).

## 4. Lifecycle

`createOpenTuiIslandController` is the shared lifecycle API. If the caller supplies a host, it uses it; if the caller supplies a `size`, it eagerly creates a Bun sidecar host; otherwise the controller starts with `host = null` and can lazily start later (`src/core/controller.ts:286-307`). When an initial island is supplied, the controller calls `setIsland()` and, if a size exists, immediately calls `syncFrame(size)` so readiness is tied to the first usable frame (`src/core/controller.ts:310-315`).

Lazy startup lives in `ensureHost(size?)`. If there is already a host, it returns it; otherwise it requires a size, calls `createOpenTuiSidecarHost`, and mounts `currentIsland` if one was already set (`src/core/controller.ts:91-111`). This means `setIsland()` can record an island without starting Bun when no host exists, while `sendKey`, `sendMouse`, `resize`, `sendCommand`, or `syncFrame` will force host creation (`src/core/controller.ts:122-145`, `src/core/controller.ts:199-207`, `src/core/controller.ts:231-246`, `src/core/controller.ts:269-278`).

Ready state is frame-based. `markReadyFrame()` stores the cached frame and marks ready only when the controller was loading; its comment says ready means “we have rendered a usable frame,” not merely that mount/update returned (`src/core/controller.ts:114-119`). `OpenTuiReadyTracker` maintains `loading | ready | error`, creates a fresh waitable promise per loading cycle, resolves it in `markReady()`, rejects it in `markError()`, and fires `onReadyStateChange`, `onReady`, or `onError` callbacks on transitions (`src/core/ready.ts:1-12`, `src/core/ready.ts:30-108`).

Disposal is explicit and two-sided. `surface.destroy()` delegates to `controller.destroy()`, which delegates to `host.destroy()` (`src/adapters/pi-tui/index.ts:368-370`, `src/core/controller.ts:281-283`). The sidecar host sends a `destroy` request, then rejects pending event waits, marks itself closed, ends/kills the child process, clears pending request/event/listener state, and returns (`src/sidecar/client.ts:451-470`, `src/sidecar/client.ts:186-193`). The server’s `destroy` method destroys the offscreen host, nulls `host` and `loadedIsland`, writes a response, and exits with code 0 (`src/sidecar/server.ts:264-272`, `src/sidecar/server.ts:299-305`). If the host process closes stdin instead of sending `destroy`, the server listens for `process.stdin` `end`, destroys the host, nulls the island, and exits (`src/sidecar/server.ts:313-319`). There is no explicit SIGTERM/SIGINT handling in `server.ts`; host-crash cleanup relies on pipe closure and the OS rather than a signal trap (`src/sidecar/server.ts:279-319`).

## 5. Pi-tui example walk-through

The Pi extension example exports `openTuiIslandPiExamples(pi: ExtensionAPI)` as the default extension entry point (`examples/pi/index.ts:425-427`). It imports Pi’s extension API types and `createEditTool`, plus pi-tui primitives such as `Text`, `matchesKey`, `Component`, and `TUI` (`examples/pi/index.ts:4-23`). The example also imports `createPiTuiSurface` and mouse-mode helpers from the built adapter output (`examples/pi/index.ts:11-15`).

It first wraps the standard edit tool. The wrapper delegates execution to `originalEdit.execute`, extracts diff details, caches the last diff in `lastDiff`, and renders either an error, “Applied,” or an `InlineDiffResult` component (`examples/pi/index.ts:429-476`). The inline result is a normal pi-tui component that renders styled diff summary lines through the host theme (`examples/pi/index.ts:151-216`).

The island overlay path is `runIslandDemo()`. It calls `ctx.ui.custom(...)` with `overlay: true`, centered width, `maxHeight: config.height + 4`, and margin (`examples/pi/index.ts:390-404`). The custom component is `OpenTuiIslandOverlay`, whose `initialize()` creates a pi-tui surface with `height: config.height`, terminal-derived `initialWidth`, `requestRender: () => tui.requestRender()`, and island module/props (`examples/pi/index.ts:303-323`). It sets `surface.focused = true`, syncs once at the initial width, updates the status, and requests a render (`examples/pi/index.ts:324-332`). Its `handleInput()` closes on Escape/q and otherwise forwards the raw input to `surface.handleInput()` (`examples/pi/index.ts:339-347`). Its `render(width)` draws a box border and embeds `surface.render(innerWidth)` inside that border (`examples/pi/index.ts:349-378`). Its `close()` destroys the surface before resolving the overlay’s `done` callback (`examples/pi/index.ts:381-386`).

Slash-command handlers are registered with `pi.registerCommand`. The example registers `opentui-counter-demo`, `opentui-diff-demo`, `opentui-editor-demo`, `review-last-diff`, and `opentui-mouse-demo` (`examples/pi/index.ts:478-573`). The counter and diff commands call `runIslandDemo`; the diff command loads diff text from args, editor text, or a built-in sample (`examples/pi/index.ts:114-149`, `examples/pi/index.ts:496-507`). The mouse command uses `ctx.ui.custom` with row/col overlay options and a `MouseIslandOverlay` that enables SGR mouse mode, sets explicit screen bounds, and forwards input (`examples/pi/index.ts:219-300`, `examples/pi/index.ts:549-573`).

The modal result example is `runPiResultEditorDemo`. It opens a centered overlay through `ctx.ui.custom`, constructs `ResultEditorOverlay`, and returns a typed `NoteEditorResultEvent` (`examples/pi/result-editor.ts:141-175`). That overlay creates a `createPiTuiSurface` with note-editor module props, focuses and syncs it, then awaits `surface.waitForEvent(isNoteEditorResultEvent)` (`examples/pi/result-editor.ts:57-74`). Inside the island, Tab emits a `save` event with text and Escape emits `cancel` through `useIslandBridge()` (`examples/islands/note-editor.island.tsx:29-52`). The Pi command receives that result; on `save`, it calls `pasteToEditor` when available and notifies success (`examples/pi/index.ts:509-525`).

## 6. The fixed-height limitation

The limitation is real and explicit. The surface options require `height: number` (`src/adapters/pi-tui/index.ts:27-35`), the surface stores it as `private readonly height`, and the constructor initializes exactly that many blank lines (`src/adapters/pi-tui/index.ts:126-148`). Every frame sync uses `height: this.height`, so OpenTUI is rendered into a fixed-height offscreen viewport (`src/adapters/pi-tui/index.ts:211-218`). Every render normalizes returned lines to the same fixed height (`src/adapters/pi-tui/index.ts:359-365`). The factory also passes `options.height` into the initial controller size (`src/adapters/pi-tui/index.ts:518-521`).

Width is dynamic; height is not. `render(width)` receives only width and can trigger a sync for a new width (`src/adapters/pi-tui/index.ts:359-365`). `sync(width = this.lastWidth)` also accepts only width and always uses `this.height` internally (`src/adapters/pi-tui/index.ts:222-234`, `src/adapters/pi-tui/index.ts:211-218`). A dynamic-height variant would need an external height source, such as terminal rows or overlay layout measurement, plus an API like `sync(width, height)` or `setSize(width, height)`. The current modal docs demonstrate caller-computed height, not auto-fill: `height: terminal.rows - 2` is passed manually (`docs/api.md:76-88`). The Pi examples also hard-code heights such as counter `height: 4`, diff `height: 13`, and editor `height: 9` (`examples/pi/index.ts:478-488`, `examples/pi/index.ts:413-421`, `examples/pi/result-editor.ts:57-67`).

## 7. In-process port feasibility

The sidecar does four concrete jobs. First, it imports an island module dynamically and validates that the requested export is either a React element or a component (`src/sidecar/server.ts:110-142`). Second, it wraps that element/component in `OpenTuiIslandBridgeProvider` so `useIslandBridge()` works inside the island (`src/sidecar/server.ts:125-170`, `src/core/bridge.tsx:49-63`). Third, it creates an offscreen OpenTUI renderer using `createTestRenderer` and a React root (`src/sidecar/offscreen-host.ts:1-9`, `src/sidecar/offscreen-host.ts:123-137`). Fourth, it translates host inputs into renderer events and returns captured frames: key input is emitted into the renderer stdin, mouse input goes through `mockMouse`, and `renderFrame()` calls `renderOnce()`, captures spans, and maps them to `HostFrame` (`src/sidecar/offscreen-host.ts:168-198`).

Conceptually, those jobs could run in-process if the host runtime could import `@opentui/core`, `@opentui/react`, and arbitrary island TSX, then call the same offscreen host functions. The code itself is not sidecar-specific: `createOffscreenOpenTuiHost` uses streams, `createTestRenderer`, React `createRoot`, and plain methods for mount/resize/focus/input/frame/destroy (`src/sidecar/offscreen-host.ts:22-31`, `src/sidecar/offscreen-host.ts:123-212`). An in-process port would replace the JSON-lines `OpenTuiHost` implementation with a direct `OpenTuiHost` implementation backed by this offscreen host.

The blocker is Node compatibility of OpenTUI, not the Surface pattern. `@opentui/core` describes itself as a TypeScript library on a native Zig core (`node_modules/@opentui/core/package.json:7-9`). It depends on `bun-ffi-structs` and optional platform-native `@opentui/core-*` packages (`node_modules/@opentui/core/package.json:47-75`). Its published JS imports `toArrayBuffer` from `bun:ffi` in the `NativeSpanFeed` bundle (`node_modules/@opentui/core/index-9y4vegye.js:3059-3068`). Its type declarations import `Pointer` from `bun:ffi` in renderer, native span feed, text buffer, and Zig structs (`node_modules/@opentui/core/renderer.d.ts:1-5`, `node_modules/@opentui/core/NativeSpanFeed.d.ts:1-11`, `node_modules/@opentui/core/text-buffer.d.ts:1-5`, `node_modules/@opentui/core/zig-structs.d.ts:1-17`). Its runtime code also uses Bun globals such as `Bun.stringWidth` in renderable code (`node_modules/@opentui/core/index-9y4vegye.js:4795-4805`, `node_modules/@opentui/core/index-9y4vegye.js:4898-4908`, `node_modules/@opentui/core/index-9y4vegye.js:10538-10540`). Its runtime plugin reads files with `Bun.file` (`node_modules/@opentui/core/index-bzjr4q9g.js:273-306`). The 3D path imports `bun-webgpu` and `bun:ffi` (`node_modules/@opentui/core/3d.js:19503-19510`).

That is why `opentui-island` keeps the Node-facing code away from OpenTUI internals. The Node client imports only Node `child_process`, `readline`, URL helpers, protocol/types, and island resolution (`src/sidecar/client.ts:1-24`). It starts Bun with Node’s `spawn(...)` rather than `Bun.spawn`, as an external process instead of importing `@opentui/core` into Node (`src/sidecar/client.ts:474-487`). The package even has a script that checks Node can import the built public entry points (`package.json:61-65`); those entry points export the client/controller/adapters, while the OpenTUI imports are confined to the sidecar implementation (`src/sidecar/offscreen-host.ts:1-9`, `src/index.ts:64-74`).

For SumoCode, a true in-process version inside Pi’s Node process would require at least one of: upstream OpenTUI Node support, a Node FFI/N-API replacement for the Bun FFI layer, or a reimplementation of the needed OpenTUI renderer/test-renderer surface. Merely copying `PiTuiOpenTuiSurface` is not enough because `controller.syncFrame()` eventually needs an `OpenTuiHost.renderFrame()` implementation that can run OpenTUI (`src/core/controller.ts:237-246`, `src/core/host.ts:10-35`). If sumo-tui itself ran under Bun rather than Pi’s Node runtime, the offscreen host could plausibly move in-process with fewer changes, but that would be a runtime change rather than a pi-tui adapter change (`README.md:16-20`, `package.json:109-113`).

If we keep the sidecar, the measurable cost is process startup plus JSON IPC. The README’s local measurements report cold start to first frame at 121.65ms average / 134.92ms p95, warm `syncFrame` at 1.30ms average / 2.08ms p95, key round-trip plus sync at 1.76ms average / 2.97ms p95, and prop update plus sync at 1.73ms average / 3.16ms p95 (`README.md:139-152`). The source does not include memory measurements, so per-sidecar memory cost is unknown from this repo. The process count cost is clear: each `createOpenTuiSidecarHost` spawns one Bun child (`src/sidecar/client.ts:474-487`), and the default `createPiTuiSurface` path creates one controller/host for one surface (`src/adapters/pi-tui/index.ts:513-534`).

## 8. Modal pattern deep dive

`createPiTuiModal` is the cleanest pattern to steal for cathedral modals, but it is lifecycle glue, not a full overlay renderer. Its options require a TUI handle, close event types, height, and island/controller options; it can enable/disable mouse, choose focus-on-open, and set close wait timeout (`src/adapters/pi-tui/index.ts:39-48`). It returns a `surface`, `result` promise, `focus()`, `waitForResult()`, `sync()`, and `destroy()` (`src/adapters/pi-tui/index.ts:52-59`).

Open lifecycle is: create the underlying surface with the TUI’s terminal columns, define a `focus()` closure that marks the surface focused and calls `tui.setFocus(surface)`, and call it unless `focusOnOpen === false` (`src/adapters/pi-tui/index.ts:412-425`). Mouse lifecycle is optional: unless `enableMouse === false`, the helper calls `attachPiTuiMouseSupport`, which enables SGR mode, adds an input listener, focuses on inside-bounds mouse events, and returns cleanup (`src/adapters/pi-tui/index.ts:383-399`, `src/adapters/pi-tui/index.ts:427-428`).

Result lifecycle is event-driven. The helper creates a promise, converts `closeOn` to a `Set`, optionally starts a timeout, and subscribes to every surface bridge event (`src/adapters/pi-tui/index.ts:429-452`). When an event type is in `closeOn`, it marks the modal settled, clears the timeout, and resolves with that event (`src/adapters/pi-tui/index.ts:452-462`). `waitForResult()` awaits that promise and always calls `destroy()` in `finally` (`src/adapters/pi-tui/index.ts:485-494`). Manual `destroy()` detaches event and mouse listeners, destroys the surface, and rejects the result if it was not already settled (`src/adapters/pi-tui/index.ts:464-482`).

Escape handling is not built into `createPiTuiModal`; it is modeled as an island event or host overlay policy. The note editor island emits `cancel` when OpenTUI reports Escape (`examples/islands/note-editor.island.tsx:47-52`). The non-helper `ResultEditorOverlay` closes on Ctrl+C or q in an error state, otherwise forwards input to the surface (`examples/pi/result-editor.ts:90-98`). The generic overlay closes on Escape/q before forwarding other keys (`examples/pi/index.ts:339-347`). Backdrop dimming is also absent from `createPiTuiModal`: the helper only returns a surface and lifecycle functions (`src/adapters/pi-tui/index.ts:485-498`), while the examples use Pi’s `ctx.ui.custom(..., { overlay: true, overlayOptions: ... })` for centered overlay placement (`examples/pi/result-editor.ts:162-174`, `examples/pi/index.ts:390-404`).

## 9. What sumo-tui should adopt

- **Surface pattern: adopt.** The `Component`/`Focusable` wrapper with cached frame lines, `requestRender`, focus-gated input, and explicit `syncFrame({ width, height })` is a good boundary between renderer and host tree (`src/adapters/pi-tui/index.ts:122-149`, `src/adapters/pi-tui/index.ts:193-234`, `src/adapters/pi-tui/index.ts:290-365`). For sumo-tui, make the size two-dimensional instead of fixed-height.
- **Frame transport: adopt concept, not necessarily JSON.** `HostFrame`/`HostLine`/`HostSpan` is a simple renderer-neutral format (`src/core/types.ts:1-30`). If in-process, skip JSON-lines and pass frames directly; if sidecar remains, the current request/response protocol is adequate (`src/sidecar/protocol.ts:14-82`, `src/sidecar/client.ts:304-335`).
- **Sidecar architecture: maybe.** Keep it for OpenTUI-on-Node today because OpenTUI’s published core imports `bun:ffi` and uses Bun globals (`node_modules/@opentui/core/index-9y4vegye.js:3059-3068`, `node_modules/@opentui/core/index-9y4vegye.js:4795-4805`). Treat it as a compatibility boundary, not a sandbox (`README.md:168-173`).
- **Modal pattern: adopt.** The event-result promise plus `waitForResult().finally(destroy)` is exactly the right host API for approval dialogs, editors, and command palettes (`src/adapters/pi-tui/index.ts:429-498`). Add our own backdrop, stacking, and Escape policy.
- **Diff algorithm: adopt or inline.** The line/span/cursor diff is small, deterministic, and useful for render invalidation (`src/core/frame-diff.ts:19-102`).
- **SGR mouse parser: adopt.** It is compact, handles scroll/move/drag/modifiers, and already produces local `HostMouseInput` after bounds translation in the adapter (`src/core/terminal-mouse.ts:28-120`, `src/adapters/pi-tui/index.ts:310-340`).

## 10. Risk + cost summary

The dependency is young: package version is `0.4.0` (`package.json:1-4`). It has a real MIT license declaration in `package.json` and the LICENSE file (`package.json:19`, `LICENSE:1-21`). Its runtime contract requires both Bun and Node (`package.json:109-113`). It also peers against specific OpenTUI and React versions (`package.json:91-97`), so OpenTUI churn is part of the maintenance surface.

Bus-factor metadata is not present in source files. External GitHub metadata checked during this spike reported one contributor (`benvinegar`) and three stargazers; the prompt’s “2 stars” value appears to have changed. That external count is not file-line-citable, while the repository URL itself is declared in `package.json` (`package.json:15-23`).

Depending on `opentui-island` buys a working Bun sidecar, Pi surface, Ink surface, modal helper, protocol, ready tracking, diffing, and mouse parsing (`docs/api.md:35-45`, `src/index.ts:64-74`). The maintenance burden is accepting one Bun child per active host plus an IPC protocol, and tracking upstream OpenTUI’s Bun/FFI internals (`src/sidecar/client.ts:474-487`, `node_modules/@opentui/core/package.json:47-75`). The safest upgrade path is to adopt the Surface/HostFrame/Modal interfaces now, keep the sidecar as a swappable `OpenTuiHost`, and leave an in-process host behind the same interface for the day OpenTUI supports Node or sumo-tui runs under Bun.
