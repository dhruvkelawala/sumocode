## 2. Root cause analysis

The issue's proposed call chain is real up to the bridge boundary: Pi's `chatContainer` is constructed as a mutable `Container` in upstream `InteractiveMode` (`node_modules/@mariozechner/pi-coding-agent/dist/modes/interactive/interactive-mode.js:228-232`), Pi adds root children in this order — header, chat, pending, status, widgets above editor, editor, widgets below editor, footer (`node_modules/@mariozechner/pi-coding-agent/dist/modes/interactive/interactive-mode.js:405-460`) — and SumoCode replaces `chatContainer.render` with `bridge.render(width)` during `installChatPagerBridge` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:792-810`). The bridge then computes an effective width, subtracts sidebar width when the sidebar is visible, computes the chat top/height from upstream chrome components, and calls `runtime.renderChatLines(lastChatWidth, lastChatHeight)` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:596-613`, `src/sumo-tui/pi-compat/sumo-interactive-mode.ts:765-780`).

The critical discrepancy is what `runtime.renderChatLines()` composites **today**. `SumoInteractiveRuntime.start()` creates a new Yoga `root`, sets it to column, creates a standalone `ChatPager`, creates a standalone `SplashTree`, creates an empty-chat quote node, and immediately calls `syncChatSlot()` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:391-408`). `syncChatSlot()` only mounts either `this.chat` or `this.splash.root` under `this.root`; it explicitly removes the empty-chat quote if present, removes splash when chat has messages, and removes chat when splash is active (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:499-518`). Therefore the current `this.root` tree for `chatContainer.render` is `root -> splash.root` in empty state or `root -> ChatPager` in active state, not top chrome + editor + footer + sidebar.

The descendants of those two states are narrow. `createSplashTree()` builds `root -> topSpacer -> content -> bottomSpacer`, with no chrome reservation or sidebar (`src/sumo-tui/cathedral/splash-tree.ts:43-55`, `src/sumo-tui/cathedral/splash-tree.ts:55-88`). `ChatPager` builds itself as a flex column with a `ScrollBox` and a `ScrolledUpBanner`, and the `ScrollBox` owns sticky-bottom scrollback inside the chat region (`src/sumo-tui/widgets/chat-pager.ts:24-55`, `src/sumo-tui/widgets/scrollbox.ts:62-90`). `renderChatLines()` lays out only `this.root` at the bridge-provided `safeWidth` and `safeHeight`, composites it into a `CellBuffer(safeHeight, safeWidth)`, converts that buffer to ANSI lines, caches by width/height/version, and returns those lines (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:429-446`). Put another way, the current active tree is `chatRoot -> ChatPager -> ScrollBox -> ChatMessage*` plus the banner, while the current empty tree is `chatRoot -> SplashTree.root -> spacer/content/spacer`; no code in `start()` attaches `topChrome`, `CathedralEditor`, `Footer`, or `StaticSidebarDock` to this runtime root (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:396-407`, `src/sumo-tui/widgets/chat-pager.ts:46-54`, `src/sumo-tui/cathedral/splash-tree.ts:61-70`, `src/sidebar.ts:343-373`).

Pi does not treat `chatContainer.render()` as an isolated replaceable region. Upstream `Container.render(width)` concatenates all child component lines into one `newLines` array (`node_modules/@mariozechner/pi-tui/dist/tui.js:61-69`). `TUI.doRender()` calls `this.render(width)` to render the full root snapshot, optionally composites overlays, extracts the cursor, and applies resets (`node_modules/@mariozechner/pi-tui/dist/tui.js:674-699`). It then diffs `newLines` against `previousLines`, detects appended lines when `newLines.length > previousLines.length`, and uses an append optimization when the first change starts at the old length (`node_modules/@mariozechner/pi-tui/dist/tui.js:764-785`). When the append target is below the previous viewport bottom, it emits `\r\n` scrolls before writing changed lines (`node_modules/@mariozechner/pi-tui/dist/tui.js:847-872`). It persists `previousLines`, `previousWidth`, `previousHeight`, and `previousViewportTop` after writing (`node_modules/@mariozechner/pi-tui/dist/tui.js:956-970`). So Pi's model is snapshot diff with append/scroll optimizations, not a pure per-chat replace.

That still explains the reported architectural failure **if** the chat slot ever returns a full shell. A full shell returned from `chatContainer.render()` is embedded between Pi's outer header and Pi's outer editor/footer because the upstream root already renders those chrome containers separately (`node_modules/@mariozechner/pi-coding-agent/dist/modes/interactive/interactive-mode.js:405-460`). As chat content grows, Pi sees the total `newLines` snapshot grow and may scroll/append the changed suffix (`node_modules/@mariozechner/pi-tui/dist/tui.js:778-785`, `node_modules/@mariozechner/pi-tui/dist/tui.js:858-872`). The splash can look correct on the first frame because first render writes the whole snapshot without clearing and has no previous frame to duplicate against (`node_modules/@mariozechner/pi-tui/dist/tui.js:699-727`). Subsequent frames reveal the bug because the nested shell is now part of chat scrollback and the outer shell remains outside it.

The full-shell tree exists elsewhere in the codebase, but it is not currently the bridge render target. `RegionRegistry` builds a complete retained Yoga shell with slots for header, chat, pending, status, widgets, editor, sidebar, footer, and overlay root (`src/sumo-tui/pi-compat/region-registry.ts:127-169`). Its adapter maps `setHeader`, `setFooter`, `setWidget`, `custom`, and `setEditorComponent` into those slots (`src/sumo-tui/pi-compat/extension-ui-adapter.ts:214-308`). The Sumo extension, meanwhile, mounts chrome through Pi's public APIs: top chrome through `setHeader`, footer through `setFooter`, editor through `setEditorComponent`, input hints through `setWidget`, and sidebar through a static dock widget (`src/extension.ts:102-111`, `src/top-chrome.ts:196-212`, `src/footer.ts:173-190`, `src/cathedral/cathedral-editor.ts:237-243`, `src/sidebar.ts:343-373`). The probable smoking gun for any failing branch is therefore not current `syncChatSlot()`, but accidentally using `RegionRegistry.root` or a future full-shell root as the object passed to `renderChatLines()`.

## 3. The proper architectural split

**Option A — chat-only root.** Keep Pi as the owner of the outer TUI root and make the bridge render exactly one thing: the chat slot. That is already the shape implied by current `SumoInteractiveRuntime.start()` and `syncChatSlot()` because `this.root` only receives splash or `ChatPager` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:396-407`, `src/sumo-tui/pi-compat/sumo-interactive-mode.ts:499-518`). The implementation should harden this by renaming `root` to `chatRoot`, adding tests that assert no shell labels appear in `chatContainer.render`, and ensuring future `RegionRegistry.root` / shell roots are never passed to `renderChatLines()`. File-level changes: `src/sumo-tui/pi-compat/sumo-interactive-mode.ts` about 80-140 LOC including renames and invariants; `src/sumo-tui/pi-compat/sumo-interactive-mode.test.ts` about 80 LOC; `test/integration/runtime-chat-scroll.test.ts` about 40 LOC; optional VHS scenario update about 30 LOC. This option preserves #51 ChatPager wiring because the bridge already routes message events into `ChatPager` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:684-750`), preserves #57/#65 sidebar because `installSidebar()` uses `StaticSidebarDock` outside the retained chat root (`src/sidebar.ts:93-135`, `src/sidebar.ts:343-373`), and preserves #60 splash behavior because splash remains the no-message chat-slot child (`src/sumo-tui/cathedral/splash-tree.ts:77-88`).

**Option B — keep full shell but force replace.** This would require a Pi API that says "the output of this child is a viewport replacement, not scrollback content." I did not find such an API in `ExtensionUIContext`: it exposes `setHeader`, `setFooter`, `setWidget`, `custom`, `setEditorComponent`, theme, editor text, and tool expansion controls, but no `replaceChat`, `clearScrollback`, or child-region invalidation primitive (`node_modules/@mariozechner/pi-coding-agent/dist/core/extensions/types.d.ts:66-186`). Pi's internal `fullRender(clear)` can clear screen and scrollback with `\x1b[2J\x1b[H\x1b[3J`, but it is private to `TUI.doRender()` and only selected by conditions like width/height changes or clear-on-shrink (`node_modules/@mariozechner/pi-tui/dist/tui.js:699-727`, `node_modules/@mariozechner/pi-tui/dist/tui.js:742-763`). A Sumo extension could write raw clear sequences, but that would fight Pi's `previousLines`/cursor bookkeeping (`node_modules/@mariozechner/pi-tui/dist/tui.js:956-970`). Option B is high risk and depends on upstream Pi changes.

**Option C — hybrid chat plus Pi overlays.** Pi already has a custom overlay API in `ExtensionUIContext.custom()` with overlay options (`node_modules/@mariozechner/pi-coding-agent/dist/core/extensions/types.d.ts:113-123`), and Pi-TUI composites overlays into the base content after rendering (`node_modules/@mariozechner/pi-tui/dist/tui.js:550-597`). Sumo's retained adapter also supports overlay and modal mounts in `RegionRegistry` (`src/sumo-tui/pi-compat/region-registry.ts:234-260`). This could host shell elements outside chat while leaving `chatContainer` pure. The risk is layout: overlays do not reserve space, while the current sidebar intentionally reserves 49 columns by wrapping root children in `StaticSidebarDock` (`src/sidebar.ts:83-91`, `src/sidebar.ts:153-167`). Overlay-only sidebar would regress wide-layout chat width and could obscure code blocks.

Recommendation: **Option A**. It matches current code, needs the smallest diff, relies on no missing Pi API, and turns the issue into an enforceable invariant: `chatContainer.render` returns chat-region lines only. Option A's visual contract is simple: in splash state there is exactly one top chrome and one input/footer outside a centered splash; in active state there is exactly one top chrome, one input frame, one footer, one optional sidebar dock at width ≥120, and the chat area contains only messages/splash. Any screenshot with `INPUT`, `READY`, `REGISTRY`, or `SUMOCODE` repeated inside chat fails the invariant.

## 4. Concrete implementation plan

**Day 1 morning — harden the runtime boundary.** In `src/sumo-tui/pi-compat/sumo-interactive-mode.ts`, rename the private `root` field to `chatRoot` everywhere used by `renderChatLines`, `syncChatSlot`, `render`, `stop`, and snapshots; if the standalone retained renderer still needs a full-screen root later, add a separate `shellRoot` field and keep it out of `renderChatLines` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:365-381`, `src/sumo-tui/pi-compat/sumo-interactive-mode.ts:429-446`, `src/sumo-tui/pi-compat/sumo-interactive-mode.ts:499-518`). Add a small private assertion such as `assertChatRootInvariant()` that checks direct children are only `chat`, `splash.root`, or the intentionally unmounted quote; this is defensive against future `RegionRegistry.root` wiring (`src/sumo-tui/pi-compat/region-registry.ts:127-169`). Estimated diff: 100 LOC. Do not change `installSidebar()`, `installTopChrome()`, `installFooter()`, or `installCathedralEditor()` because they already mount through Pi's outer UI APIs (`src/extension.ts:102-111`).

Pseudocode for the fixed relationship:

```ts
class SumoInteractiveRuntime {
  private chatRoot: SumoNode | undefined;
  private shellRoot: SumoNode | undefined; // future standalone/full-retained only
  private chat: ChatPager | undefined;
  private splash: SplashTree | undefined;

  async start() {
    this.chatRoot = new SumoNode(yoga.Node.create());
    this.chatRoot.flexDirection = FLEX_DIRECTION_COLUMN;
    this.chat = ChatPager.create(yoga, undefined, controls);
    this.splash = createSplashTree(yoga, undefined, snapshot);
    this.syncChatSlot();
  }

  renderChatLines(width, chatHeight) {
    syncChatSlot();
    chatRoot.width = width;
    chatRoot.height = chatHeight;
    layout(chatRoot, width, chatHeight);
    return ansi(composite(chatRoot, new CellBuffer(chatHeight, width)));
  }
}
```

## 5. Edge cases and risk register

- Splash → first message transition. Severity: high. The only legal chat-slot children are splash before messages and `ChatPager` after messages; `syncChatSlot()` currently performs exactly that removal/addition (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:499-518`). Mitigation: keep the splash test that verifies the quote disappears after first message (`src/sumo-tui/pi-compat/sumo-interactive-mode.test.ts:94-108`).
- ChatPager sticky-bottom during streaming. Severity: high. `appendToLast()` updates estimated height, marks streaming mode, and schedules render (`src/sumo-tui/widgets/chat-pager.ts:74-88`); `ScrollBox.notifyContentChanged()` sticks to bottom unless the user manually scrolled (`src/sumo-tui/widgets/scrollbox.ts:136-149`). Mitigation: add a streaming long-code-block test.
- Sidebar dock at width ≥120 versus narrow. Severity: medium. The bridge narrows chat by 49 columns when sidebar is visible (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:596-613`), and `StaticSidebarDock` reserves the same 49 columns plus gutter (`src/sidebar.ts:108-135`). Mitigation: test at 119 and 120 columns.
- Long code blocks. Severity: high. Pi crashes if any final line exceeds terminal width (`node_modules/@mariozechner/pi-tui/dist/tui.js:876-907`). Mitigation: fake-provider/VHS scenario with fenced code and a 260-character row.
- Theme switching mid-session. Severity: medium. Upstream `setTheme()` requests a render when successful (`node_modules/@mariozechner/pi-coding-agent/dist/modes/interactive/interactive-mode.js:1562-1570`). Mitigation: assert cache invalidation after theme change or call `runtime.requestRender()` through existing controls (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:420-427`).
- Resize during render. Severity: medium. `renderChatLines()` caches by width/height/version (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:429-446`) and upstream clears on width/height changes (`node_modules/@mariozechner/pi-tui/dist/tui.js:742-754`). Mitigation: force resize in a unit test and expect no stale shell tokens.
- Memory with 200+ messages. Severity: medium. `ChatPager` virtualizes beyond `DEFAULT_MAX_RENDERED_MESSAGES = 200` and archives earlier messages behind a placeholder (`src/sumo-tui/widgets/chat-pager.ts:20-20`, `src/sumo-tui/widgets/chat-pager.ts:170-201`). Mitigation: regression test with 250 messages.
- Compositor cache invalidation. Severity: medium. `requestRender()` increments `frameVersion` before delegating to Pi or the scheduler (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:420-427`, `src/sumo-tui/pi-compat/sumo-interactive-mode.ts:495-497`). Mitigation: ensure every ChatPager mutation schedules render.
- Frame scheduler interaction. Severity: low/medium. The scheduler coalesces frames and uses 16ms streaming ticks (`src/sumo-tui/runtime/frame-scheduler.ts:28-43`, `src/sumo-tui/runtime/frame-scheduler.ts:102-145`). Mitigation: don't bypass `requestRender()` when changing root ownership.
- `pi update` followed by no restart. Severity: medium. The CLI dynamically imports `SUMO_TUI_MODULE` or package `@dhruvkelawala/sumocode/sumo-interactive-mode` once when constructing the interactive mode (`node_modules/@mariozechner/pi-coding-agent/dist/main.js:593-596`). Mitigation: document restart requirement and smoke with `./bin/sumocode.sh` to avoid installed stale code (`bin/sumocode.sh:7-18`).

## 7. Fall-back plan

Smallest mitigation if Option A becomes invasive: add a guard in `renderChatLines()` that strips or refuses obvious shell output. For example, after rendering, if lines contain the editor border label, footer state label, or registry sidebar label, log a diagnostic and return only the rows between computed chat bounds. This is a hide-the-symptom fallback, not a fix, because those shell labels currently come from Sumo extension components mounted through Pi APIs (`src/top-chrome.ts:199-210`, `src/footer.ts:176-188`, `src/cathedral/cathedral-editor.ts:237-243`, `src/sidebar.ts:343-373`).

A second mitigation is to force Pi full redraw/clear after each chat render. Pi's private `fullRender(clear)` can clear screen and scrollback with `\x1b[2J\x1b[H\x1b[3J` (`node_modules/@mariozechner/pi-tui/dist/tui.js:699-727`), but `ExtensionUIContext` does not expose a public clear-scrollback method (`node_modules/@mariozechner/pi-coding-agent/dist/core/extensions/types.d.ts:66-186`). Writing raw clear bytes from Sumo would desynchronize Pi's `previousLines` and cursor state (`node_modules/@mariozechner/pi-tui/dist/tui.js:956-970`). Use only for local diagnosis.

Worst case: disable `SUMO_TUI=1` / the `SumoInteractiveMode` path and run on raw Pi-TUI while keeping the public extension chrome. The upstream extension APIs can still mount header, footer, widgets, and editor (`node_modules/@mariozechner/pi-coding-agent/dist/modes/interactive/interactive-mode.js:1522-1558`), and Sumo's extension already uses them (`src/extension.ts:102-111`). This would lose retained ChatPager/mouse-scroll work, but it would eliminate any chance that a retained full-shell root is embedded inside `chatContainer.render`.

Operational fallback for a release branch: ship a feature flag such as `SUMO_TUI_CHAT_PAGER=0` that skips `installChatPagerBridge()` while leaving Sumo's public extension hooks active. The wrapper currently enables Sumo TUI globally via `SUMO_TUI=1` and loads the custom interactive mode through `SUMO_TUI_MODULE` (`bin/sumocode.sh:7-18`), while the bridge itself is installed in `configureUpstreamBeforeInit()` (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:931-934`). A flag around that one install site would restore upstream `chatContainer.render` without removing top chrome, footer, editor, input hints, or sidebar extension code (`src/extension.ts:102-111`). This fallback is safer than raw terminal clearing because it keeps Pi's `previousLines` and viewport bookkeeping internally consistent (`node_modules/@mariozechner/pi-tui/dist/tui.js:956-970`).

Hot fix candidate: if the implementing agent finds a branch where `renderChatLines()` composites a `RegionRegistry.root` or a `createSidebarTree().root`, change that one call to composite the chat slot only. `RegionRegistry.root` is demonstrably full shell (`src/sumo-tui/pi-compat/region-registry.ts:127-169`), while current `syncChatSlot()` demonstrates the intended chat-only mount (`src/sumo-tui/pi-compat/sumo-interactive-mode.ts:499-518`).
