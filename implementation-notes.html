SumoCode startup implementation notes

Running notes for the startup-time work on the sumocode launcher.

2026-05-19 baseline

Used the existing scripts/perf-startup.mjs node-pty benchmark. Baseline on commit 3f84472 chore(pi): update backend to 0.75.3:

  • launcher-dry-run: ~40.5ms average middle runs
  • print-mode: ~6.78s average middle runs
  • first-frame: ~3.79s average middle runs

Debug diagnostics show roughly 1.5s before extension diagnostics install, then about 1.0s in the jiti import of src/sumo-tui/pi-compat/sumo-interactive-mode.ts, then retained runtime reaches eager splash paint quickly.

Decision: optimize launcher work first

The largest remaining first-frame cost is upstream Pi startup plus jiti importing the retained runtime. The safest low-risk win is reducing repeated shell work in bin/sumocode.sh, especially the retained-TUI patch probe used on every sumocode launch and dry run. Bigger changes like precompiling the retained runtime would conflict with the repo's no-build-step convention and need a separate design decision.

Change: project-local Pi main fast path

bin/sumocode.sh now resolves node_modules/@earendil-works/pi-coding-agent/dist/main.js directly when using the project-local node_modules/.bin/pi. This avoids grepping the pnpm shell shim on the hot path. Manual timing of bin/sumocode.sh --dry-run --offline --no-extensions --no-session improved from roughly 40–50ms to roughly 25–30ms after the first filesystem-cold run.

Change: benchmark cleanup

scripts/perf-startup.mjs now settles first-frame samples only once and sends a follow-up termination signal after capturing the first retained-frame marker. The previous benchmark could print results but keep node-pty handles alive long enough for the wrapper command to time out.

Post-change snapshot

Ran node scripts/perf-startup.mjs:

  • launcher-dry-run: 26.6ms average middle runs
  • print-mode: 6963.9ms average middle runs
  • first-frame: 3756.9ms average middle runs

Tradeoff: the launcher improvement is intentionally narrow and low risk. It does not hide that print mode and first-frame time remain dominated by upstream Pi startup/model/session work and retained-runtime TypeScript import cost.

Follow-up: diagnose instead of pre-painting

Launcher pre-paint was reverted. Added early process diagnostics and used sumocode -d traces to split startup into Pi process/module load, SumoCode extension import, retained runtime import, and actual splash paint.

Findings: with the extension loaded, first visible splash was about 4.45s in Dhruv's trace. Without -e src/extension.ts, the same retained runtime reached splash around 1.98s, so the SumoCode extension import graph was a major contributor.

Changes: deferred heavy post-splash imports from sumo-interactive-mode.ts; removed a top-chrome static import of the retained runtime; deferred heavy tool/command/sidebar registration until after session_start. The benchmark reported first-frame around 2.42s at this point. Remaining cost is mostly Pi baseline plus retained runtime import; further cuts likely require deeper runtime splitting or upstream Pi startup work.

Second cut: deferred ChatPager/notification/shell-transition imports until immediately after the eager splash paint. A 3-run startup snapshot reported first-frame at 2.19s avg middle runs (min 2.12s, max 2.23s). This moves visible splash closer to the observed ~1.98s Pi+SumoTUI baseline without the SumoCode extension.
